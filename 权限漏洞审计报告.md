## 二、漏洞详情与修复方案

### 🔴 漏洞 1：越权查看个人资料

| 项目 | 内容 |
|:-----|:------|
| **漏洞描述** | `/profile` 接口接收 URL 参数 `user_id`，但**未验证用户是否已登录**，也未验证当前用户是否有权查看该 user_id 的资料。任何人只需修改 URL 中的 user_id 即可查看任意用户的邮箱、手机号、余额等隐私信息。 |
| **攻击场景** | 1. 攻击者访问 `http://localhost:5000/profile?user_id=1`<br>2. 看到管理员的邮箱、手机、余额<br>3. 修改 URL 为 `?user_id=2` 查看 alice 的资料<br>4. 无需登录，即可遍历所有用户隐私 |
| **影响等级** | 🟠 高 |
| **修复方案** | 添加登录检查 + 所有权验证：`user_id` 必须等于当前会话的 `user_id` |
| **验证方法** | 未登录访问 `/profile` 应跳转登录页；登录后访问 `?user_id=其他ID` 应提示无权查看 |

**修复前代码（存在漏洞）：**
```python
@app.route('/profile')
def profile():
    user_id = request.args.get('user_id', type=int)
    # ❌ 没有登录检查
    # ❌ 没有所有权验证
    c.execute("SELECT * FROM users WHERE id=?", (user_id,))
    return render_template('profile.html', user=user)
```

**修复后代码（安全）：**
```python
@app.route('/profile')
def profile():
    if 'username' not in session:          # ✅ 登录检查
        flash('请先登录', 'warning')
        return redirect(url_for('login'))
    user_id = request.args.get('user_id', type=int)
    if user_id != session.get('user_id'):   # ✅ 所有权验证
        flash('无权查看其他用户的资料', 'error')
        return redirect(url_for('index'))
    c.execute("SELECT * FROM users WHERE id=?", (user_id,))
    return render_template('profile.html', user=user)
```

### 🟠 漏洞 2：越权充值

| 项目 | 内容 |
|:-----|:------|
| **漏洞描述** | `/recharge` 接口从表单接收 `user_id` 和 `amount`，但**未验证当前登录用户是否是该 user_id 的拥有者**。攻击者可以给任意账号充值或扣款（配合负值漏洞可盗刷余额）。 |
| **攻击场景** | 1. 攻击者以普通用户身份登录（user_id=2）<br>2. 提交充值表单，`user_id=1`（管理员），`amount=-99999`<br>3. 管理员余额被扣 99999<br>4. 攻击者可清空任意用户余额 |
| **影响等级** | 🟠 高 |
| **修复方案** | 验证 `user_id` 必须等于当前会话的 `user_id` |
| **验证方法** | 修改充值表单中的 `user_id` 为他人ID，应提示"无权为其他用户充值" |

**修复前代码（存在漏洞）：**
```python
@app.route('/recharge', methods=['POST'])
def recharge():
    user_id = request.form.get('user_id', type=int)
    amount = request.form.get('amount', type=float)
    # ❌ 没有登录检查
    # ❌ 没有所有权验证
    # ❌ 没有金额正负检查
    c.execute("UPDATE users SET balance = balance + ? WHERE id=?", (amount, user_id))
```

**修复后代码（安全）：**
```python
@app.route('/recharge', methods=['POST'])
def recharge():
    if 'username' not in session:          # ✅ 登录检查
        flash('请先登录', 'warning')
        return redirect(url_for('login'))
    user_id = request.form.get('user_id', type=int)
    amount = request.form.get('amount', type=float)
    if user_id != session.get('user_id'):   # ✅ 所有权验证
        flash('无权为其他用户充值', 'error')
        return redirect(url_for('index'))
    if amount <= 0:                         # ✅ 金额正负检查
        flash('充值金额必须为正数', 'error')
        return redirect(url_for('profile', user_id=user_id))
    c.execute("UPDATE users SET balance = balance + ? WHERE id=?", (amount, user_id))
```

### 🔴 漏洞 3：负值充值（盗刷漏洞）

| 项目 | 内容 |
|:-----|:------|
| **漏洞描述** | `/recharge` 接口**未对金额 `amount` 做正负校验**。攻击者可以提交负值（如 `-500`），导致 SQL 执行 `balance = balance + (-500)`，相当于从目标账户扣款。 |
| **攻击场景** | 1. 攻击者提交 `amount=-99999`<br>2. SQL 执行：`balance = balance + (-99999)`<br>3. 目标余额减少 99999<br>4. 实际效果：从他人账户"盗刷"余额 |
| **影响等级** | 🔴 严重 |
| **修复方案** | 添加金额必须大于 0 的校验 |
| **验证方法** | 充值 `-500`，应提示"充值金额必须为正数" |

**修复前：**
```python
# ❌ 无金额校验
c.execute("UPDATE users SET balance = balance + ? WHERE id=?", (amount, user_id))
# amount=-500 ➜ balance = balance - 500（扣款成功）
```

**修复后：**
```python
if amount <= 0:                             # ✅ 金额必须为正数
    flash('充值金额必须为正数', 'error')
    return redirect(url_for('profile', user_id=user_id))
# 负值在 SQL 执行前被拦截
```
